## 执行步骤

### 步骤一：获取日志

使用 AskUserQuestion 确认日志来源：

- 用户直接粘贴日志内容
- 从文件读取（提供路径）
- 从应用控制台获取（指导用户如何获取）

如果用户提供了文件路径，使用 Read 读取日志文件。

### 步骤二：分类与解析

逐行分析日志内容：

1. **识别来源**: Main Process / Renderer / MCP Server / Network
2. **分类严重度**: Error / Warning / Info
3. **提取时间戳**: 如果有，用于时间线重建
4. **提取错误类型**: 错误名称、错误码、堆栈信息

### 步骤三：关联分析

1. 按时间戳排序，重建事件链
2. 跨进程关联（例如 MCP 崩溃 → IPC 超时 → UI 错误）
3. 区分根因和级联效应（找到链条中第一个错误）

### 步骤四：代码定位

使用 Grep 在项目源码中搜索：

1. 搜索错误消息文本，定位抛出错误的源码位置
2. 搜索相关的 handler、service、component
3. 阅读相关代码理解上下文

### 步骤五：输出分析报告

```
🔬 日志分析报告

━━━ 摘要 ━━━
发现: X 个错误, Y 个警告
来源: [进程列表]
严重度: 严重/高/中/低

━━━ 错误详情 ━━━
❌ [ERR-001] 错误标题
   时间: timestamp
   来源: Main/Renderer/MCP
   日志: 相关日志行
   根因: 解释
   修复:
     1. 具体步骤
     2. 具体步骤
   文件: 涉及的源码路径

━━━ 事件时间线 ━━━
time [Source] 事件描述
time [Source] ❌ 错误发生
time [Source] 级联影响

━━━ 修复建议 ━━━
1. 立即修复: ...
2. 防御加固: ...
3. 补充诊断: ...
```

如果日志信息不完整，明确指出还需要哪些信息（来源、类型、获取方式）。
