# 项目技术决策记录

## 架构决策

### ADR-001: 使用 Electron 作为桌面应用框架
**日期**: 项目初期
**状态**: 已采纳

**背景**：
需要构建跨平台桌面 AI 客户端应用

**决策**：
选择 Electron 框架

**理由**：
- 跨平台支持（macOS, Windows, Linux）
- 丰富的生态系统
- 可以使用 Web 技术栈（React, TypeScript）
- 成熟的社区和工具链

### ADR-002: 使用 Zustand 进行状态管理
**日期**: 项目初期
**状态**: 已采纳

**背景**：
需要轻量级的状态管理方案

**决策**：
使用 Zustand 替代 Redux

**理由**：
- API 简洁，学习曲线低
- 性能优秀（基于订阅）
- TypeScript 支持良好
- 内置持久化中间件
- 无需 Provider 包裹

### ADR-003: Main Process 作为单一数据源
**日期**: 2026-03
**状态**: 已采纳

**背景**：
多窗口应用需要状态同步

**决策**：
Main Process 维护所有业务状态，Renderer 通过 IPC 同步

**理由**：
- 避免状态不一致
- 简化多窗口同步逻辑
- 便于持久化和恢复
- 符合 Electron 最佳实践

### ADR-004: 使用 Tailwind CSS
**日期**: 项目初期
**状态**: 已采纳

**背景**：
需要高效的样式解决方案

**决策**：
使用 Tailwind CSS v4

**理由**：
- 开发效率高
- 样式一致性好
- 构建体积小（按需生成）
- 易于维护和定制

### ADR-005: 环境变量管理 API 配置
**日期**: 2026-03-07
**状态**: 已采纳

**背景**：
API base_url 硬编码在代码中，不便于部署

**决策**：
将 API 配置移至环境变量

**实施**：
- super-client-r: `MAIN_VITE_CONFIG_API_BASE_URL`
- node-auth: `API_BASE_URL`

**理由**：
- 支持多环境部署
- 提高安全性
- 便于配置管理
